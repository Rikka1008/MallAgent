# LLM 上下文驱动售后 Agent 设计

## 目标

将售后 Agent 从“关键词规则路由 + 固定模板回复”升级为“LLM 负责理解与规划、后端负责受控执行”。用户可以用省略表达连续追问，例如“我买的什么”“我想退了”，系统能够结合最近对话、已选订单和工具结果回答。

## 边界

- LLM 可以：理解用户语义、读取近期会话状态、选择已有子 Agent、决定需要补充的槽位、根据工具结果生成中文回复。
- 子 Agent 可以：调用已注册的订单、物流、退款、售后、政策工具。
- 后端必须：从登录 Cookie 获取会员身份；校验订单归属、参数完整性、售后资格；为写操作提供幂等键；拒绝未授权或不符合条件的执行。
- LLM 不可以：伪造工具结果、绕过校验、直接访问 Mall、直接确认退款或创建售后单。

## 方案对比

1. 纯提示词 Agent：实现快，但模型可能绕过校验、编造结果或重复执行写操作；不采用。
2. 规则路由 + 模板回复：可控但上下文理解差，当前问题即由此造成；不采用。
3. LLM 规划 + 受控执行器：LLM 输出结构化计划，后端校验后调用子 Agent，再由 LLM 基于真实结果生成回复；采用。

## 数据流

1. 读取 LangGraph 短期状态：最近消息、当前意图、槽位、订单候选、上一轮工具结果。
2. 读取长期记忆：用户近期订单和偏好摘要；只检索与本轮相关的语义记忆。
3. LLM 输出结构化 `AgentPlan`：`intent`、`agent_name`、`slot_updates`、`required_slots`、`needs_tool`、`reply_goal`。
4. 后端合并槽位并校验计划是否合法；只允许调用注册表中的子 Agent。
5. 子 Agent 调用 Mall 工具，工具以当前已认证会员身份执行。
6. LLM 接收已脱敏的状态和真实工具结果，生成中文回复；无法执行时生成明确的补充信息提示。
7. 保存更新后的结构化状态；对写操作记录幂等键和审计事件。

## 上下文策略

- 短期上下文：最近 8 轮用户/助手消息，外加当前结构化槽位；超过窗口时保留结构化摘要。
- 工具结果：只保留当前订单、商品、物流、退款、售后申请的必要字段，禁止把认证信息或原始 token 放入模型上下文。
- 长期记忆：用于偏好和历史订单提示，不作为订单归属或售后资格的依据。

## 失败与安全策略

- LLM 超时、输出不合法或置信不足：返回“暂时无法理解”的中文提示，不自动执行写操作。
- LLM 请求未注册 Agent 或缺失关键参数：拒绝该计划，并提示用户补充信息。
- 工具查询为空或校验失败：把真实失败原因交给回复生成器，不允许模型改写为成功。
- 创建售后申请：必须包含订单、商品、类型、原因，并以用户 ID + 订单 + 商品 + 类型 + 原因生成幂等键。

## 验收标准

- “查询订单 → 我买的什么”能引用该订单的商品名称和数量。
- “查询物流 → 选择订单 → 我想退了”能复用订单，追问商品和原因。
- LLM 不可调用未注册 Agent，不可越过订单归属校验。
- 工具错误、模型超时、无效结构化输出均有中文可理解回复。
- 订单查询、物流查询、退货续接、写操作校验均有自动化测试。
